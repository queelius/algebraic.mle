# JOSS Audit Report: algebraic.mle

## Summary
- **Status**: NEEDS WORK
- **Package**: algebraic.mle v1.1.0 (R)
- **Paper exists**: Yes (938 words in body -- within 750-1750 range)
- **Bibliography exists**: Yes (10 entries, all cited, no orphans)
- **Reviewer checklist**: 18/25 items pass

The package has a well-written paper, solid software design, CRAN publication, and
4+ years of development history. However, three issues require attention before
submission: (1) the complete absence of a formal test suite is a critical blocker,
(2) there is no CONTRIBUTING.md file, and (3) there is no R CMD check CI workflow.
The paper itself is in good shape with only minor issues.

---

## Critical Gaps (Blockers)

### 1. No formal test suite
- **What**: The package has no `tests/` directory. The DESCRIPTION declares
  `Config/testthat/edition: 3`, but no testthat infrastructure or test files exist.
  Validation currently relies solely on R CMD check (which exercises examples and
  vignettes) and manual vignette rendering.
- **Why it blocks**: JOSS requires "An automated test suite" as an explicit checklist
  item. Reviewers will flag this immediately. JOSS editors have historically rejected
  reviews where the only testing is through examples.
- **Fix**: Create `tests/testthat.R` and `tests/testthat/` with test files covering
  at minimum:
  - `test-mle.R`: Constructor, `params()`, `vcov()`, `confint()`, `se()`, `aic()`,
    `loglik_val()`, `sampler()`, `marginal()`, `expectation()`, `mse()`, `bias()`,
    `coef()`, `logLik()`, `nobs()`, `summary()`, `print()`
  - `test-mle_numerical.R`: Wrapping `optim()` output, Hessian inversion, convergence flag
  - `test-mle_boot.R`: Wrapping `boot::boot()` output, bootstrap CIs, bias estimation
  - `test-mle_rmap.R`: Delta method and Monte Carlo transformations
  - `test-mle_weighted.R`: Inverse-variance weighting, single-element edge case
  - `test-utils.R`: `confint_from_sigma()`

  Aim for at least 80% line coverage. Run `covr::package_coverage()` to verify.

## Major Issues (Should Fix)

### 2. No CONTRIBUTING.md
- **What**: There is no contribution guidelines file at the repository root.
- **Source**: Community audit
- **Fix**: Create `CONTRIBUTING.md` with at minimum: how to report bugs (link to
  GitHub Issues), how to propose changes (fork + PR workflow), coding style
  conventions, and how to run tests. The `usethis::use_tidy_contributing()` template
  is a good starting point.

### 3. No CODE_OF_CONDUCT.md
- **What**: There is no code of conduct file at the repository root.
- **Source**: Community audit
- **Fix**: Run `usethis::use_code_of_conduct(contact = "lex@metafunctor.com")` or
  manually add the Contributor Covenant (the standard for JOSS submissions).

### 4. No R CMD check CI workflow
- **What**: The only GitHub Actions workflow is `pkgdown.yaml` for building the
  documentation site. There is no `R-CMD-check.yaml` workflow to run automated
  checks on push/PR. JOSS reviewers look for evidence of CI.
- **Source**: Software audit
- **Fix**: Add `.github/workflows/R-CMD-check.yaml` using the standard
  `r-lib/actions` template. Run `usethis::use_github_action("check-standard")` or
  copy from https://github.com/r-lib/actions/tree/v2/examples.

### 5. README does not mention CRAN installation
- **What**: The README only documents GitHub installation via `devtools::install_github()`.
  Since the package is on CRAN, the primary installation method should be
  `install.packages("algebraic.mle")`.
- **Source**: Software audit / Community audit
- **Fix**: Add a CRAN installation section at the top of the Installation section:
  ```r
  install.packages("algebraic.mle")
  ```

### 6. CITATION.cff version is outdated
- **What**: `CITATION.cff` lists `version: "1.0.0"` but the package is at v1.1.0.
- **Source**: Community audit
- **Fix**: Update CITATION.cff to version 1.1.0.

### 7. Paper State of the Field could be stronger on comparison
- **What**: The State of the Field section names five competing packages (`stats4::mle`,
  `bbmle`, `maxLik`, `mle.tools`, `MLE`) with citations -- this is good and above
  average. However, the section could be strengthened by explicitly noting what
  `algebraic.mle` does that none of the others do (a comparison table or bullet list
  of features vs. alternatives). Currently the differentiation paragraph at the end
  is prose-only.
- **Source**: Field scout analysis
- **Fix**: Consider adding a brief feature comparison table or a more structured
  differentiation summary. This is a "should fix" because JOSS reviewers focus
  heavily on State of the Field.

## Specialist Reports

### Software Auditor

**Source files**: 7 R files in `R/` totaling ~595 lines of code (excluding comments/blanks).

**Architecture**: Clean S3 class hierarchy. Base `mle` class with three subclasses
(`mle_numerical`, `mle_boot`, `rmap_mle`) plus a combiner (`mle_weighted`). All
subclasses properly inherit from `mle` via the `superclasses` parameter. Method
dispatch is correct -- 35 S3 methods registered in NAMESPACE.

**API completeness**: All methods documented in the paper (`params`, `vcov`, `confint`,
`se`, `aic`, `loglik_val`, `sampler`, `pred`, `expectation`, `marginal`, `rmap`,
`bias`, `mse`, `observed_fim`, `score_val`, `coef`, `logLik`, `nobs`) are exported
and have roxygen documentation with `@examples`.

**Dependencies**: 6 imports (`algebraic.dist`, `stats`, `boot`, `mvtnorm`, `MASS`,
`numDeriv`) -- all on CRAN, reasonable and lightweight.

**Findings**:
1. **CRITICAL**: No `tests/` directory despite `Config/testthat/edition: 3` in DESCRIPTION
2. **MAJOR**: No R-CMD-check CI workflow (only pkgdown deployment)
3. **MINOR**: `testthat` not in Suggests despite config declaration
4. **MINOR**: `rmap.mle()` uses `stopifnot(is.integer(n), ...)` but the default
   `n = 1000L` passes; user calling `rmap(x, g, n = 1000)` (without L suffix) would
   fail. Consider `n = 1000` (numeric) and coerce internally.
5. **MINOR**: `mle_weighted()` calls `nobs()` on each element, but `nobs()` may
   return NULL for manually constructed `mle` objects -- this would fail in `sum()`.

### Field Scout

**Domain**: Statistical computing / maximum likelihood estimation in R

**Competing packages identified in paper**:
1. `stats4::mle` (base R) -- correctly characterized as S4-coupled
2. `bbmle` (Ben Bolker) -- correctly noted as extending stats4 with mle2
3. `maxLik` (Henningsen & Toomet) -- correctly positioned as optimization-focused
4. `mle.tools` (Mazucheli et al.) -- correctly scoped as bias-correction specific
5. `MLE` (Tsagris & Piperaki) -- correctly noted as distribution-catalog approach

**Additional packages the paper could consider mentioning**:
- `fitdistrplus` (Delignette-Muller & Dutang, JSS 2015): Widely used for fitting
  distributions by MLE. While its scope is distribution fitting rather than algebraic
  MLE manipulation, it is the package most R users encounter when searching for MLE
  tools. JOSS reviewers familiar with R statistics will likely ask why it is not mentioned.
- `likelihood` (Murphy, CRAN): Older package for likelihood-based inference.
- `stats::optim` itself is discussed but not as a competitor (correct, since
  algebraic.mle wraps its output rather than replacing it).

**Differentiation assessment**: The paper's core claim -- that algebraic.mle is a
*post-estimation algebra* rather than an optimizer or distribution catalog -- is
well-supported and genuinely distinct from all five named competitors. The "missing
middle" framing (between raw optim and formula-based model fitting) is compelling and
accurate.

**Downstream usage evidence**: The paper claims use by `maskedcauses`, `maskedhaz`,
and `compositional.mle`. These packages exist in the author's r-universe and GitHub
repositories, confirming the claimed dependency chain.

**Overall field assessment**: The State of the Field is above average for a JOSS
submission. Five packages are named with proper citations, and each is accurately
characterized. The main gap is `fitdistrplus`, which a reviewer will likely flag.
